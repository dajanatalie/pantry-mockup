A scheduled Power Automate flow checks stock against the reorder point daily and creates an order proposal when stock falls below it. On unreliable data, automation degrades to assisted ordering and human verification.

New Supply Request (Power Apps form)

Illustrative approval thresholds — configurable as data, not hard-coded: < €100 auto-approved · €100–499 → Office Manager · ≥ €500 → Cost Center Owner. Actual values are decided by the business owner.

Approval & order queue (Power Automate)

#ItemQtyValueApproverStatus

Approving an order generates the structured supplier email and sets the order to Ordered. Confirm receipt adds the delivered quantity to stock and writes the audit entry — closing the loop: request → approval → order → receipt → inventory update.

    High-level target architecture — Power Platform

    1 · Experience

    Power Apps (Canvas)Request submission · inventory view · approval status — target implementation of this UX concept
    Read-only Pantry AssistantRead-only assistant grounded on validated inventory and order data (Copilot Studio). AI assists, but does not order or modify records.
    Teams / OutlookNotifications: approvals, low stock, order status
    ▼

    2 · Automation — Power Automate

    Flow 1 · Low-stock detectionScheduled daily · stock < reorder point → order proposal
    Flow 2 · Approval routingThreshold-based: auto / Office Mgr / Cost Center Owner — thresholds configurable as data
    Flow 3 · Supplier orderStructured PO email with order table & PO number
    Flow 4 · Receipt & stock updateConfirmation → stock += qty · audit entry
    ▼

    3 · Data — SharePoint lists

    InventoryItem, stock, reorder point, unit cost, location, last verified
    Requests / OrdersFull lifecycle incl. status history
    SuppliersContact, items, lead time
    Config & AuditApproval thresholds as data · controlled audit and status history (versioned, restricted access)

    Location-aware data model reduces future rework. Multi-site rollout requires a separate operating-model and platform assessment. Documented migration path to Dataverse when scale or complexity requires it.

    ▼

    4 · Integration

    Outlook (standard connector)Email-based supplier communication — matches supplier maturity; confirmation by reply referencing the PO number
    Microsoft Entra IDIdentity & role-based access for requesters, approvers, admins
    Connection identityDedicated connection or service identity where permitted by organizational policy
    ▼

    5 · Governance

    EnvironmentsDev → Test → Prod
    ALMSolutions + deployment pipelines
    DLP policiesAllowed connectors defined
    Approval logic as dataBusiness owners change thresholds — not developers
    Backlog & knowledge baseUser stories in Jira (Design vs. Config) · thresholds, rule owners & supplier info documented in Confluence
    Assumptions and trade-offs: SharePoint possible for a faster MVP, subject to existing tenant licensing and security requirements · email-based supplier communication (integration would be over-engineering at current supplier maturity) · assistant is read-only in the MVP. Phase 2 (deliberately deferred): write access, autonomous actions, recommendations, demand forecasting, and more complex conversational scenarios — enabled only once data quality and governance gates are met.
